# Major Features

## Project Execution

Orchestrates an AI coding agent through repeated development iterations until all requirements in a project file are marked as passing, then opens a pull request with the completed work. This is Ralph's core purpose: turning a YAML requirements file into merged code without human intervention.

**User Interface:** Run `ralph <project.yaml>` (locally with `--local`, or remotely on Kubernetes by default).

- A project must have a non-empty name and at least one requirement to be considered valid.
- Each iteration selects the highest-priority failing requirement and directs the AI agent to implement only that one requirement at a time.
- The loop continues until every requirement has `passing: true`, up to a configurable maximum number of iterations (default: 10); if the maximum is reached with failures remaining, execution stops with an error.
- If the AI agent writes a `blocked.md` file indicating it cannot make progress, the loop halts immediately rather than continuing to iterate.
- Billing or quota errors from the AI provider are detected and immediately stop the loop to avoid wasted spend.
- Each successful iteration is committed with a message derived from the agent's `report.md`, or auto-generated from the diff if no report was written.
- A pull request is opened only when there are commits ahead of the base branch; if all requirements were already passing, no PR is created.

## Service Management

Starts and stops developer services (e.g., databases, local servers) that the AI agent needs running during development iterations. This ensures the agent can run tests against real dependencies and that those services are cleanly shut down when work is done.

**User Interface:** Configured via the `services` list in `.ralph/config.yaml`; services are started and stopped automatically around `ralph <project.yaml>` runs.

- Services are started in declared order before the iteration loop begins and stopped in reverse order after the loop completes.
- A service with a configured port is polled via a TCP connection until it is ready to accept connections, up to a configurable timeout (default: 30 seconds).
- If a service fails to start, an AI agent is invoked to diagnose and attempt to fix the problem before giving up.
- Log files generated by services are cleaned up after each iteration.

## Remote Workflow Execution

Submits development jobs to a Kubernetes cluster via Argo Workflows so that long-running AI development work does not need to remain on the developer's machine. This allows teams to run multiple projects in parallel on shared infrastructure.

**User Interface:** Default mode when running `ralph <project.yaml>` without `--local`; use `--watch` to follow logs live. Manage running jobs with `ralph list` and `ralph stop`.

- Remote submission is rejected if the current branch has local commits that have not been pushed to the remote, ensuring the workflow container can access the latest code.
- Each workflow runs in an isolated container with the project file, credentials, and configuration mounted from Kubernetes Secrets and ConfigMaps.
- Active workflows can be listed and stopped by name.

## GitHub Webhook Automation

Listens for GitHub events on pull requests and automatically triggers Ralph workflows in response to comments or approvals, enabling a fully automated review-and-merge cycle driven by collaborators' PR interactions.

**User Interface:** The `ralph-webhook` HTTP service receives `POST /webhook` events from GitHub.

- Every incoming webhook must carry a valid HMAC-SHA256 signature computed with the per-repository secret; requests without a valid signature are rejected.
- Events from the bot's own GitHub account and from any configured ignored users are silently discarded to prevent feedback loops.
- If an allowlist is configured for a repository, only events from users on that list are processed; all others are dropped.
- A pull request review with state `approved` triggers an Argo merge workflow.
- A pull request comment or a review with a non-empty comment body triggers an Argo development workflow that has the AI respond to or act on the comment.
- Only events associated with pull requests are processed; comments on plain issues are ignored.

## PR Comment Response

Runs the AI agent in response to a specific pull request comment body, allowing collaborators to request changes, ask questions, or steer the implementation directly from the GitHub PR interface.

**User Interface:** Triggered automatically by the webhook service, or invoked directly with `ralph comment <body> --repo --branch --pr`.

- The AI agent is given the comment text as its primary instruction and may make code changes, push commits, or reply via additional context in the project file.
- Custom instructions for comment handling can be provided via `.ralph/comment-instructions.md`, overriding the built-in defaults.

## PR Merge

Verifies that all project requirements are passing and then merges the pull request, cleaning up the associated project file. This closes the development loop by ensuring no incomplete work is merged.

**User Interface:** Run `ralph merge <branch> --pr <N>`, or triggered automatically by the webhook service when a PR is approved.

- The AI agent confirms that every requirement in the project is marked as passing before the merge proceeds.
- After a successful merge, the project YAML file is removed from the repository to keep the projects directory clean.
- Custom instructions for the merge check can be provided via `.ralph/merge-instructions.md`.

## Code Review

Runs the AI agent against a set of configured review standards and produces findings as a new project file, then opens a pull request so the findings can be tracked and addressed. This automates the enforcement of team coding standards and architectural guidelines.

**User Interface:** Run `ralph review` to execute all configured review items.

- Each review item must specify its standard via exactly one source: an inline text snippet, a local file path, or a URL.
- Review items configured with the `domain-function` loop are expanded into one review pass per domain function listed in `architecture.yaml`, so every function is checked individually against the standard.
- The order in which review items are evaluated is randomized each run to prevent systematic bias in which items receive attention first.

## Architecture Generation

Analyzes the codebase and produces an `architecture.yaml` file that documents the apps, features, domain functions, and modules in the repository. This gives Ralph and team members a machine-readable map of the project structure.

**User Interface:** Run `ralph review architecture`.

- The generated file is validated against a strict schema: every app must have a name, description, entry point, and at least one feature; every feature must reference at least one function by file and name; every module must declare whether it is a `domain` or `implementation` module.
- If validation fails, the AI agent is given up to three attempts to fix the errors before giving up.
- The final validated file is committed and a pull request is opened for team review.

## Project Validation

Checks a project YAML file for structural correctness before it is submitted for execution, giving authors fast feedback on mistakes without consuming AI resources.

**User Interface:** Run `ralph validate <project.yaml>`.

- A project is rejected if it has no name.
- A project is rejected if it contains no requirements.

## Credential and Infrastructure Configuration

Stores API keys, GitHub tokens, and webhook configuration into Kubernetes Secrets and ConfigMaps so that remote Ralph workflows and the webhook service can authenticate with external services without embedding credentials in source code.

**User Interface:** Run `ralph config github`, `ralph config opencode`, `ralph config pulumi`, `ralph config webhook`, or `ralph config webhook-secret`.

- GitHub personal access tokens are stored as a Kubernetes Secret for use by remote workflows.
- All AI provider credentials found in the local OpenCode auth file are extracted and stored as a Kubernetes Secret.
- Per-repository HMAC webhook secrets are generated, registered with GitHub, and stored as a Kubernetes Secret so that the webhook service can verify incoming event signatures.
- Webhook service configuration (repository list, allowed users, image settings) is stored as a Kubernetes ConfigMap.
